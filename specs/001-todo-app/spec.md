## Clarifications

### Session 2026-05-15

- Q: 라우팅 앱에서 좌측 네비게이션을 어떻게 공유하나? → A: 통합 단일 SideNav —
  메인·캘린더·통계 세 화면이 같은 SideNav를 공유한다. 뷰 항목(받은편지함/오늘/예정/
  지연/완료)은 /main의 해당 뷰로, 캘린더·통계는 각 라우트로, 카테고리는 /main 필터로
  이동한다.
- Q: 통계 화면 데이터는 live인가 고정인가? → A: 전부 고정 합성값 — KPI·주간 막대·
  도넛·히트맵·카테고리별 진행률 모두 디자인 번들과 동일한 정적 값. 카테고리별
  진행률은 원본 샘플 할 일 상수(SAMPLE_TASKS) 분포 기준으로 계산하며 사용자의 live
  localStorage 데이터를 반영하지 않는다.
- Q: 다크/라이트 테마 토글을 어디에 두나? → A: 공유 SideNav 하단(footer). 메인·
  캘린더·통계에서 접근 가능하다. 로그인 화면에는 토글을 두지 않고 현재 테마만 적용된다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 메인 화면에서 할 일을 관리한다 (Priority: P1)

사용자는 메인 화면의 가운데 리스트에서 오늘 할 일을 한눈에 보고, 새 할 일을 빠르게
추가하고, 완료 체크하고, 좌측 네비게이션으로 뷰(받은편지함/오늘/예정/지연/완료)와
카테고리를 전환한다. 이것이 앱의 핵심 사용 흐름이다.

**Why this priority**: 할 일 추가·완료·뷰 전환이 동작하지 않으면 Todo 앱으로서의
가치가 없다. 이 한 가지만 구현해도 독립적으로 쓸 수 있는 MVP다.

**Independent Test**: 메인 화면만 띄운 상태에서 할 일을 추가하고, 완료 체크하고,
뷰를 전환하며 각 뷰에 올바른 할 일만 보이는지 확인하면 완결적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 메인 화면이 "오늘" 뷰로 열려 있을 때, **When** 상단 추가 입력란에 제목을
   입력하고 Enter를 누르면, **Then** 새 할 일이 리스트 맨 위에 나타나고 입력란이 비워진다.
2. **Given** 리스트에 미완료 할 일이 있을 때, **When** 그 할 일의 체크박스를 클릭하면,
   **Then** 할 일이 완료 처리되어 취소선이 표시되고 "오늘" 뷰에서 사라지며 "완료" 뷰에
   나타난다.
3. **Given** "오늘" 뷰에 마감일이 지난 할 일이 섞여 있을 때, **When** 화면을 보면,
   **Then** 지연된 할 일은 "지연" 그룹, 오늘 마감 할 일은 "오늘" 그룹으로 나뉘어 표시된다.
4. **Given** 좌측 네비게이션이 보일 때, **When** "예정" 뷰를 선택하면, **Then** 미래
   마감 할 일만 날짜별 그룹으로 표시된다.
5. **Given** 좌측 네비게이션에 카테고리 목록이 있을 때, **When** 특정 카테고리를 선택하면,
   **Then** 해당 카테고리의 미완료 할 일만 리스트에 표시된다.
6. **Given** 리스트 상단에 필터 칩(전체/우선순위 높음/즐겨찾기)이 있을 때, **When**
   "우선순위 높음" 칩을 누르면, **Then** 현재 뷰에서 우선순위가 높은 할 일만 남는다.
7. **Given** 현재 뷰에 표시할 할 일이 하나도 없을 때, **When** 화면을 보면, **Then**
   "할 일이 없습니다" 빈 상태 메시지가 표시된다.

### User Story 2 - 할 일의 상세 정보를 편집한다 (Priority: P2)

사용자는 리스트에서 할 일을 선택해 우측 상세 패널에서 제목과 메모를 편집하고,
우선순위·카테고리를 인라인 팝오버로 바꾸고, 즐겨찾기/삭제하고, 서브태스크
체크리스트를 추가·체크·삭제한다.

**Why this priority**: 할 일을 "관리"하려면 세부 편집이 필요하다. P1 위에 얹히는
핵심 기능이지만, P1만으로도 MVP가 성립하므로 P2다.

**Independent Test**: 리스트에서 할 일 하나를 선택해 상세 패널을 열고, 제목/메모를
바꾸고, 우선순위·카테고리를 변경하고, 서브태스크를 추가·체크·삭제한 뒤 변경이
리스트에도 반영되는지 확인하면 검증된다.

**Acceptance Scenarios**:

1. **Given** 리스트에서 할 일을 클릭했을 때, **When** 우측 상세 패널을 보면, **Then**
   그 할 일의 제목·마감일·우선순위·카테고리·서브태스크·메모가 표시된다.
2. **Given** 상세 패널이 열려 있을 때, **When** 제목 또는 메모를 수정하면, **Then**
   변경이 즉시 가운데 리스트의 해당 행에도 반영된다.
3. **Given** 상세 패널의 우선순위 항목을 클릭했을 때, **When** 팝오버에서 다른
   우선순위를 선택하면, **Then** 우선순위가 바뀌고 리스트 행의 우선순위 표시가 갱신된다.
4. **Given** 상세 패널의 카테고리 항목을 클릭했을 때, **When** 팝오버에서 다른
   카테고리를 선택하면, **Then** 카테고리가 바뀌고 리스트/네비게이션 카운트가 갱신된다.
5. **Given** 상세 패널에 서브태스크 입력란이 있을 때, **When** 텍스트를 입력하고
   Enter를 누르면, **Then** 서브태스크가 목록에 추가되고 진행률 표시가 갱신된다.
6. **Given** 서브태스크가 있을 때, **When** 서브태스크 체크박스를 토글하면, **Then**
   완료 개수와 진행률 막대가 갱신되고, 리스트 행의 서브태스크 진행 표시도 갱신된다.
7. **Given** 상세 패널이 열려 있을 때, **When** 삭제 버튼을 누르면, **Then** 할 일이
   리스트에서 제거되고 상세 패널은 "선택된 할 일이 없습니다" 빈 상태로 돌아간다.
8. **Given** 상세 패널 또는 리스트 행에 즐겨찾기 버튼이 있을 때, **When** 즐겨찾기를
   토글하면, **Then** 별표 상태가 바뀌고 "즐겨찾기" 필터 칩 결과에 반영된다.

### User Story 3 - 캘린더에서 일정을 본다 (Priority: P2)

사용자는 캘린더 화면에서 월간 그리드로 할 일을 카테고리 색상 도트와 함께 보고,
이전/다음/오늘 버튼으로 달을 이동하고, 날짜를 클릭해 우측 미리보기 패널에서 그 날의
할 일을 확인한다.

**Why this priority**: 일정 조망은 중요한 가치지만 할 일 관리(P1) 없이는 보여줄 게
없다. 메인 화면과 같은 데이터를 다른 시각으로 보여주는 독립 화면이다.

**Independent Test**: 캘린더 화면을 띄워 2026년 5월이 6주 그리드로 나오고, 할 일이
있는 날에 도트가 보이고, 오늘(5/15)이 강조되고, 날짜 클릭 시 우측 패널이 그 날의
할 일을 보여주며, 이전/다음 버튼으로 달이 바뀌는지 확인하면 검증된다.

**Acceptance Scenarios**:

1. **Given** 캘린더 화면이 열렸을 때, **When** 화면을 보면, **Then** 2026년 5월이
   일요일 시작 6주(42칸) 그리드로 표시되고 오늘(5/15)이 강조된다.
2. **Given** 마감일이 있는 할 일이 존재할 때, **When** 그리드를 보면, **Then** 해당
   날짜 칸에 카테고리 색상 도트와 함께 할 일 제목이 표시되고, 3개를 넘으면 "+N개 더"가
   표시된다.
3. **Given** 캘린더 그리드가 보일 때, **When** 임의의 날짜 칸을 클릭하면, **Then** 그
   칸이 선택 표시되고 우측 미리보기 패널이 그 날짜의 할 일 목록(없으면 빈 메시지)을
   보여준다.
4. **Given** 캘린더 상단에 이전/다음/오늘 네비게이션이 있을 때, **When** 다음 버튼을
   누르면, **Then** 표시 월이 6월로 바뀌고, "오늘" 버튼을 누르면 5월로 돌아온다.

### User Story 4 - 통계 대시보드를 확인한다 (Priority: P3)

사용자는 통계 화면에서 완료율·완료 개수·연속 달성·활발한 시간 KPI 카드, 주간 완료
막대 차트, 카테고리 도넛 차트, 카테고리별 진행률 바, 20주 활동 히트맵을 본다.

**Why this priority**: 동기 부여용 부가 화면이다. 핵심 할 일 관리 흐름과 독립적이며
우선순위가 낮다.

**Independent Test**: 통계 화면을 띄워 4개 KPI 카드, 주간 막대 차트, 도넛 차트, 진행률
바, 히트맵이 모두 렌더링되고, 기간 토글(이번 주/달/분기/연간)이 선택 상태를 바꾸는지
확인하면 검증된다.

**Acceptance Scenarios**:

1. **Given** 통계 화면이 열렸을 때, **When** 화면을 보면, **Then** 완료율/완료한 할
   일/연속 달성/가장 활발한 시간 4개 KPI 카드가 표시된다.
2. **Given** 통계 화면이 열렸을 때, **When** 화면을 보면, **Then** 주간 완료 막대
   차트, 카테고리 도넛 차트, 카테고리별 진행률 바, 20주 활동 히트맵이 모두 렌더링된다.
3. **Given** 상단에 기간 토글(이번 주/이번 달/분기/연간)이 있을 때, **When** 다른
   기간을 선택하면, **Then** 선택된 토글이 활성 상태로 바뀐다.

---

### User Story 5 - 로그인 화면으로 진입한다 (Priority: P3)

사용자는 로그인 화면에서 좌측 다크 히어로(데모 프리뷰 리스트)와 우측 폼(Google/GitHub
소셜 버튼, 이메일/비밀번호 입력, 로그인 상태 유지 체크박스)을 보고, 로그인하여 앱
메인으로 진입한다.

**Why this priority**: 첫 진입 화면이지만 실제 인증 로직 없이 표현 위주다. 핵심 할 일
관리와 독립적이다.

**Independent Test**: 로그인 화면을 띄워 좌/우 분할 레이아웃, 데모 프리뷰, 소셜 버튼,
폼 필드, 체크박스가 디자인대로 렌더링되고, 로그인 버튼/소셜 버튼을 누르면 메인 화면으로
이동하는지 확인하면 검증된다.

**Acceptance Scenarios**:

1. **Given** 앱에 처음 진입했을 때, **When** 로그인 화면을 보면, **Then** 좌측 다크
   히어로 + 우측 폼 분할 레이아웃이 표시된다.
2. **Given** 로그인 화면이 열렸을 때, **When** "로그인 상태 유지" 체크박스를 클릭하면,
   **Then** 체크 상태가 토글된다.
3. **Given** 로그인 화면이 열렸을 때, **When** 로그인 버튼 또는 소셜 로그인 버튼을
   누르면, **Then** 메인 화면("오늘" 뷰)으로 이동한다.

---

### User Story 6 - 데이터가 유지되고 테마를 바꾼다 (Priority: P2)

사용자가 만든 모든 변경(추가·수정·완료·삭제·서브태스크)은 브라우저에 저장되어
새로고침해도 유지되며, 사용자는 라이트/다크 테마를 토글할 수 있고 그 선택도 유지된다.

**Why this priority**: 영속화가 없으면 모든 상호작용이 새로고침에 사라져 실사용이
불가능하다. P1·P2 스토리를 가로지르는 기반 가치라 P2로 둔다.

**Independent Test**: 할 일을 추가/수정한 뒤 페이지를 새로고침해 변경이 남아 있는지,
테마를 다크로 바꾼 뒤 새로고침해 다크가 유지되는지 확인하면 검증된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 할 일을 추가·수정·완료·삭제했을 때, **When** 페이지를
   새로고침하면, **Then** 모든 변경사항이 그대로 유지된다.
2. **Given** 앱을 처음 실행했을 때(저장된 데이터 없음), **When** 메인 화면을 보면,
   **Then** 업무 테마 샘플 할 일(회의/디자인/개발 등)이 시드되어 표시된다.
3. **Given** 테마 토글 컨트롤이 있을 때, **When** 다크로 전환하면, **Then** 4개 화면
   전체가 다크 테마로 바뀌고, 새로고침 후에도 다크가 유지된다.

---

### Edge Cases

- 빈 제목으로 할 일/서브태스크를 추가하려 하면(공백만 입력) 추가되지 않는다.
- 현재 상세 패널에서 보고 있는 할 일을 삭제하면 패널은 빈 상태로 돌아간다.
- localStorage에 데이터가 없거나 손상되어 읽을 수 없으면 샘플 데이터로 안전하게
  초기화한다.
- 캘린더에서 한 날짜에 할 일이 4개 이상이면 3개만 보이고 "+N개 더"로 요약한다.
- "오늘" 뷰는 마감일이 오늘이거나 지난 미완료 할 일을 모두 포함한다(지연 항목 포함).
- 마감일이 없는 할 일은 날짜 기반 뷰(예정/지연)에는 나타나지 않지만 받은편지함에는
  나타난다.
- 모든 할 일을 삭제하면 각 뷰는 빈 상태 메시지를 표시한다.
- 캘린더에서 다른 달로 이동한 뒤 "오늘" 버튼을 누르면 2026년 5월로 돌아온다.

## Requirements *(mandatory)*

### Functional Requirements

#### 공통 / 내비게이션

- **FR-001**: 시스템은 로그인, 메인, 캘린더, 통계 4개 화면을 제공해야 한다.
- **FR-002**: 메인·캘린더·통계 화면은 **하나의 통합 SideNav**를 공유해야 한다. SideNav의
  뷰 항목(받은편지함/오늘/예정/지연/완료)은 메인 화면의 해당 뷰로, 캘린더·통계 항목은
  각 라우트로, 카테고리 항목은 메인 화면의 해당 카테고리 필터로 이동시켜야 한다.
  로그인 화면은 이 네비게이션 밖의 별도 진입 화면이어야 한다.
- **FR-003**: 시스템은 공유 SideNav 하단(footer)에 라이트/다크 테마 전환 토글을
  제공해야 하고, 테마는 4개 화면 전체에 적용되어야 한다. 로그인 화면에는 토글을 두지
  않으며 현재 테마만 적용한다.
- **FR-004**: 시스템은 "오늘"을 2026-05-15로 고정해 상대 날짜 계산(오늘/내일/어제/N일
  후·전)과 샘플 데이터를 결정론적으로 유지해야 한다.
- **FR-005**: 시스템은 핸드오프 번들의 디자인 토큰(`colors_and_type.css`)과 시각
  스타일(`todo.css`)을 그대로 사용해 4개 화면을 시각적으로 충실히 재현해야 한다.

#### 데이터 / 영속화

- **FR-006**: 시스템은 할 일 데이터를 브라우저 localStorage에 영속화해야 하며,
  새로고침 후에도 사용자의 모든 변경이 유지되어야 한다.
- **FR-007**: 시스템은 저장된 데이터가 없는 최초 실행 시 업무 테마 샘플 할 일(회의/
  디자인/개발/기획/개인 카테고리)을 시드해야 한다.
- **FR-008**: 시스템은 저장된 데이터가 손상되어 읽을 수 없을 때 샘플 데이터로 안전하게
  초기화해야 한다(앱이 깨지지 않아야 한다).
- **FR-009**: 시스템은 선택한 테마 설정도 영속화해야 한다.

#### 메인 화면 — 리스트 & 뷰 (US1)

- **FR-010**: 사용자는 리스트 상단 입력란에 제목을 입력하고 Enter로 새 할 일을 추가할
  수 있어야 하며, 추가 후 입력란은 비워져야 한다.
- **FR-011**: 시스템은 공백만 입력된 할 일 추가 요청을 무시해야 한다.
- **FR-012**: 사용자는 할 일의 체크박스로 완료/미완료를 토글할 수 있어야 한다.
- **FR-013**: 사용자는 할 일의 별표 버튼으로 즐겨찾기를 토글할 수 있어야 한다.
- **FR-014**: 시스템은 받은편지함/오늘/예정/지연/완료 5개 뷰를 제공하고, 각 뷰는
  정의된 규칙에 맞는 할 일만 표시해야 한다:
  - 받은편지함 = 미완료 전체
  - 오늘 = 마감이 오늘이거나 지난 미완료
  - 예정 = 마감이 미래인 미완료
  - 지연 = 마감이 지난 미완료
  - 완료 = 완료된 전체
- **FR-015**: 사용자는 좌측 네비게이션에서 카테고리를 선택해 해당 카테고리의 미완료
  할 일만 필터링할 수 있어야 한다.
- **FR-016**: "오늘" 뷰는 할 일을 "지연" 그룹과 "오늘" 그룹으로 나눠 표시해야 한다.
- **FR-017**: "예정" 뷰는 할 일을 마감일별 그룹으로 나눠 표시해야 한다.
- **FR-018**: 시스템은 전체 / 우선순위 높음 / 즐겨찾기 필터 칩을 제공하고, 칩 선택 시
  현재 뷰 결과를 추가로 필터링해야 한다.
- **FR-019**: 좌측 네비게이션의 각 뷰·카테고리는 해당하는 할 일 개수를 표시해야 하며,
  데이터 변경 시 카운트가 갱신되어야 한다.
- **FR-020**: 시스템은 현재 뷰에 표시할 할 일이 없을 때 빈 상태 메시지를 표시해야 한다.
- **FR-021**: 각 할 일 행은 마감일(상대 표기), 카테고리(색상 도트+이름), 서브태스크
  진행(완료/전체), 메모 유무를 메타 정보로 표시해야 한다.

#### 메인 화면 — 상세 패널 (US2)

- **FR-022**: 사용자는 리스트에서 할 일을 클릭해 우측 상세 패널에서 그 할 일을 선택할
  수 있어야 한다.
- **FR-023**: 선택된 할 일이 없으면 상세 패널은 빈 상태를 표시해야 한다.
- **FR-024**: 사용자는 상세 패널에서 할 일의 제목과 메모를 편집할 수 있어야 하고,
  변경은 리스트에 즉시 반영되어야 한다.
- **FR-025**: 사용자는 상세 패널의 인라인 팝오버로 우선순위(높음/보통/낮음/없음)를
  변경할 수 있어야 한다.
- **FR-026**: 사용자는 상세 패널의 인라인 팝오버로 카테고리를 변경할 수 있어야 한다.
- **FR-027**: 사용자는 상세 패널에서 할 일을 삭제할 수 있어야 하며, 삭제된 할 일이
  현재 선택 항목이면 패널은 빈 상태로 돌아가야 한다.
- **FR-028**: 사용자는 상세 패널에서 서브태스크를 추가(Enter)·완료 토글·삭제할 수
  있어야 한다.
- **FR-029**: 시스템은 서브태스크가 있을 때 완료 개수와 진행률(%)을 진행 막대로
  표시하고, 변경 시 갱신해야 한다.
- **FR-030**: 시스템은 공백만 입력된 서브태스크 추가 요청을 무시해야 한다.

#### 캘린더 화면 (US3)

- **FR-031**: 시스템은 월간 그리드를 일요일 시작 6주(42칸)로 표시해야 한다.
- **FR-032**: 시스템은 마감일이 있는 할 일을 해당 날짜 칸에 카테고리 색상 도트와 함께
  표시하고, 한 칸에 4개 이상이면 3개만 표시하고 "+N개 더"로 요약해야 한다.
- **FR-033**: 시스템은 오늘 날짜와 현재 달이 아닌 날짜를 시각적으로 구분해야 한다.
- **FR-034**: 사용자는 이전/다음 버튼으로 달을 이동하고 "오늘" 버튼으로 2026년 5월로
  돌아올 수 있어야 한다.
- **FR-035**: 사용자는 날짜 칸을 클릭해 선택할 수 있어야 하고, 우측 미리보기 패널이
  선택 날짜의 할 일 목록(없으면 빈 메시지)을 표시해야 한다.

#### 통계 화면 (US4)

- **FR-036**: 시스템은 완료율, 완료한 할 일 수, 연속 달성, 가장 활발한 시간 4개 KPI
  카드를 표시해야 한다.
- **FR-037**: 시스템은 주간 완료 막대 차트, 카테고리 도넛 차트, 카테고리별 진행률 바,
  20주 활동 히트맵을 표시해야 한다. 통계 화면의 모든 수치(KPI·주간·도넛·히트맵·
  카테고리별 진행률)는 디자인 번들과 동일한 고정 합성값을 사용하며, 사용자의 live
  localStorage 데이터를 반영하지 않는다.
- **FR-038**: 시스템은 기간 토글(이번 주/이번 달/분기/연간)을 제공하고 선택 상태를
  반영해야 한다.

#### 로그인 화면 (US5)

- **FR-039**: 시스템은 좌측 다크 히어로(데모 프리뷰 리스트)와 우측 폼으로 구성된
  분할 레이아웃 로그인 화면을 표시해야 한다.
- **FR-040**: 로그인 폼은 Google/GitHub 소셜 버튼, 이메일/비밀번호 입력, "로그인 상태
  유지" 체크박스, 로그인 버튼을 포함해야 한다.
- **FR-041**: 사용자가 로그인 버튼 또는 소셜 로그인 버튼을 누르면 메인 화면("오늘"
  뷰)으로 이동해야 한다(실제 인증 검증은 범위 밖).

#### 접근성 (Constitution Principle V)

- **FR-042**: 모든 상호작용 컨트롤은 시맨틱 요소와 접근 가능한 이름(보이는 텍스트
  또는 aria-label)을 가져야 한다.
- **FR-043**: 사용자는 키보드만으로 할 일 추가, 완료 토글, 서브태스크 추가를 수행할
  수 있어야 하고, 포커스 상태가 시각적으로 드러나야 한다.

### Testing Requirements *(mandatory — Constitution Principle I)*

> 본 프로젝트는 TDD가 비협상 원칙이다. `/speckit-tasks`의 "테스트 선택" 기본값을
> 무효화하고, 아래 테스트를 명시적으로 요구한다.

- **TR-001**: 할 일 store의 순수 로직(뷰 필터링, 지연/오늘/예정 판정, 뷰·카테고리
  카운트, 상대 날짜 포맷, 추가/수정/삭제/토글/서브태스크 변환)은 구현 전에 단위
  테스트가 작성되어야 하고, 실패를 확인한 뒤 구현해야 한다.
- **TR-002**: 인터랙티브 컴포넌트 동작(할 일 추가, 완료 토글, 즐겨찾기 토글, 뷰 전환,
  필터 칩, 상세 패널 편집, 우선순위·카테고리 팝오버, 서브태스크 CRUD, 캘린더 날짜
  선택·달 이동, 테마 토글)은 구현 전에 행위 테스트가 작성되어야 한다.
- **TR-003**: localStorage 영속화·시드·손상 복구 동작은 테스트로 검증되어야 한다.
- **TR-004**: 모든 테스트는 Red(실패)→Green(통과)→Refactor 순서를 따르며, 한 번도
  실패한 적 없는 테스트는 신뢰하지 않는다.

### Key Entities

- **Task (할 일)**: 하나의 할 일. 속성 — 고유 id, 제목, 마감일, 우선순위(높음/보통/
  낮음/없음), 카테고리 참조, 즐겨찾기 여부, 완료 여부, 메모, 서브태스크 목록.
- **Subtask (서브태스크)**: 한 Task에 속한 체크리스트 항목. 속성 — 고유 id, 텍스트,
  완료 여부.
- **Category (카테고리)**: 색상 태그. 속성 — 고유 id, 이름, 색상. 고정 5종(디자인/
  개발/회의/기획/개인).
- **View (뷰)**: 할 일 목록을 거르는 관점. 받은편지함/오늘/예정/지연/완료 5종.
- **Priority (우선순위)**: 높음/보통/낮음/없음 4단계, 각 표시 색상 보유.
- **ThemePreference (테마 설정)**: 라이트/다크. 영속화 대상.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 메인 화면에서 제목 입력과 Enter만으로 5초 이내에 새 할 일을
  추가할 수 있다.
- **SC-002**: 5개 뷰와 5개 카테고리 필터 각각이 정의된 규칙에 맞는 할 일만 100%
  정확하게 표시한다.
- **SC-003**: 페이지 새로고침 후에도 사용자가 만든 모든 변경(추가/수정/완료/삭제/
  서브태스크/테마)이 100% 유지된다.
- **SC-004**: 4개 화면 모두 라이트·다크 테마에서 핸드오프 디자인 번들과 시각적으로
  일치한다(레이아웃·색·간격·타이포).
- **SC-005**: 캘린더에서 임의의 날짜를 클릭하면 그 날짜의 할 일이 우측 패널에 정확히
  표시된다.
- **SC-006**: 사용자는 마우스 없이 키보드만으로 할 일 추가·완료 토글·서브태스크
  추가를 완료할 수 있다.
- **SC-007**: store 순수 로직과 인터랙티브 컴포넌트 동작에 대한 모든 테스트가
  존재하고 통과하며, 각 테스트는 구현 전 실패가 관찰된 이력을 가진다.
- **SC-008**: 최초 실행 시 업무 테마 샘플 할 일이 시드되어 사용자가 빈 화면을 보지
  않는다.

## Assumptions

- 단일 사용자 앱이며 다중 사용자·권한·실제 인증은 범위 밖이다. 로그인 화면은 표현
  위주이고, 로그인/소셜 버튼은 메인 화면으로 라우팅만 한다.
- 백엔드·서버·외부 API는 없다. 모든 상태는 클라이언트와 localStorage에 존재한다.
- "오늘"은 디자인 번들과 동일하게 2026-05-15로 하드코딩한다. 실제 시스템 시각을 쓰지
  않는다.
- 카테고리는 디자인 번들의 고정 5종을 사용하며, 사용자 정의 카테고리 추가/편집은 범위
  밖이다(네비게이션의 카테고리 추가 버튼은 시각적 표현으로만 존재).
- 통계 화면의 모든 수치(KPI·주간 막대·도넛·히트맵·카테고리별 진행률)는 디자인
  번들과 동일한 고정 합성값을 사용한다. 카테고리별 진행률은 원본 샘플 할 일
  상수(SAMPLE_TASKS) 분포 기준으로 계산하며, 사용자가 만든 live 할 일은 반영하지
  않는다(실시간 집계는 범위 밖, 디자인 충실도 우선).
- 검색·더보기·복제·알림 등 디자인에 아이콘만 있고 동작이 정의되지 않은 보조 컨트롤은
  시각적으로 존재하되 비기능 상태로 둔다(디자인 충실도 우선, YAGNI).
- 캘린더의 "새 할 일"/"이 날에 할 일 추가" 버튼은 사용자가 명시한 캘린더 범위에
  포함되지 않으므로, 디자인대로 렌더링하되 비기능 상태로 둔다. 할 일 생성의 표준
  흐름은 메인 화면이다.
- 프로토타입의 디자인 캔버스(pan/zoom 래퍼)와 Tweaks 패널 호스트 프로토콜은 프로덕션
  구현에 포함하지 않는다. 테마 전환은 일반 UI 컨트롤로 대체한다.
- 데스크톱 뷰포트를 1차 타깃으로 한다. 모바일 단일 컬럼 반응형은 범위 밖이다.
