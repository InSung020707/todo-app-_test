# Feature Specification: 사용자 인증과 클라우드 동기화 (Auth & Cloud Sync)

**Feature Branch**: `002-supabase-auth`
**Created**: 2026-05-22
**Status**: Draft
**Input**: User description: "현재 Todo 앱에 실제 로그인 기능과 클라우드 저장소를 추가. 이메일/비밀번호 회원가입·로그인, Google 소셜 로그인, 사용자별 할 일 격리, 비로그인 상태에서 메인 접근 시 로그인 화면으로 이동, 로그아웃 버튼. 목표: 친구에게 링크를 공유하여 각자 자신의 할 일을 관리. 품질: User Story별 E2E 시나리오, 기존 기능 회귀 테스트, 접근 제어 E2E 테스트 포함."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 회원가입과 개인 할 일 관리 (Priority: P1)

새로운 사용자가 이메일과 비밀번호로 계정을 만들고, 로그인 후 자기만의 할 일 목록을
만들고 관리합니다. 다른 사용자로 로그인하면 그 사용자의 할 일만 보이며, 서로의
데이터는 절대로 섞이지 않습니다.

**Why this priority**: 이 하나의 기능으로 "친구에게 링크를 보내 각자 자기 할 일을
관리한다"는 프로젝트 목표가 달성됩니다. 다른 기능(로그아웃, Google 로그인)은 편의
기능이며, 이 핵심 기능이 동작하지 않으면 나머지가 모두 의미를 잃습니다.

**Independent Test**: 서로 다른 두 이메일로 가입하여 각각 다른 할 일을 추가한 뒤,
번갈아 로그인하여 자기 데이터만 보이는지 검증할 수 있습니다. 이 흐름 하나만으로
"개인용 클라우드 Todo"라는 MVP 가치가 전달됩니다.

**Acceptance Scenarios**:

1. **Given** 가입되지 않은 이메일을 가진 새 방문자가 로그인 화면에 있을 때,
   **When** 이메일과 비밀번호로 회원가입을 완료하면,
   **Then** 즉시 로그인 상태로 메인 화면에 진입한다(별도 로그인 절차 없이).
2. **Given** 등록된 사용자가 로그인 화면에 있을 때,
   **When** 올바른 이메일/비밀번호로 로그인하면,
   **Then** 본인의 할 일 목록이 표시된 메인 화면으로 이동한다.
3. **Given** 사용자 A가 로그인된 상태에서,
   **When** 할 일을 추가/완료/별표/삭제하거나 하위 작업을 추가/삭제하면,
   **Then** 변경 사항이 클라우드에 저장되어 페이지 새로고침 후에도 유지된다.
4. **Given** 사용자 A의 할 일이 클라우드에 저장되어 있을 때,
   **When** 같은 브라우저에서 로그아웃 후 사용자 B로 로그인하면,
   **Then** 사용자 B는 본인의 할 일만 보이고 사용자 A의 할 일은 어떤 화면(메인,
   캘린더, 통계)에서도 보이지 않는다.
5. **Given** 잘못된 비밀번호를 입력했을 때,
   **When** 로그인을 시도하면,
   **Then** "이메일 또는 비밀번호가 올바르지 않습니다" 형태의 한국어 안내가 표시되고
   화면 전환은 일어나지 않는다.

**E2E Test Scenarios** (기능 회귀 포함):

- **회원가입 → 로그인 → 할 일 관리 전체 흐름**: 신규 가입 → 자동 로그인 상태로 메인
  진입 → 할 일 추가/수정/완료/별표/삭제, 하위 작업 추가/삭제, 뷰 전환
  (today/upcoming/starred), 캘린더 진입, 통계 진입, 테마 토글까지 모두 동작 확인 →
  새로고침 → 모든 데이터 유지 확인.
- **기존 기능 회귀**: 인증 도입 후에도 기존 할 일 CRUD, 하위 작업 CRUD, 별표/완료
  토글, 필터링, 캘린더, 통계, 테마 토글의 행동이 변하지 않음을 확인.
- **데이터 격리**: 사용자 A로 가입/로그인하여 할 일 N개 추가 → 로그아웃 → 사용자 B로
  가입/로그인 → 사용자 B의 어떤 화면에도 A의 할 일이 0개로 보임을 확인.

### User Story 2 - 인증 가드와 로그아웃 (Priority: P2)

로그인하지 않은 사람이 메인 화면 또는 인증이 필요한 다른 화면 주소로 직접 접근하면
자동으로 로그인 화면으로 이동합니다. 사용자는 언제든 로그아웃할 수 있으며, 로그아웃
이후에는 같은 브라우저에서도 자동 로그인되지 않아 다시 로그인이 요구됩니다.

**Why this priority**: 보안과 공용 컴퓨터 사용 시나리오를 위한 필수 기능입니다.
US1만으로는 비로그인 사용자가 URL을 직접 입력해 메인에 접근할 수 있어 데이터 보호의
틈이 생깁니다. 다만 US1이 동작하지 않으면 US2는 의미가 없으므로 우선순위는 P2.

**Independent Test**: 시크릿 브라우저 창에서 메인 URL을 직접 입력하여 로그인 화면으로
리다이렉트되는지, 로그인 후 로그아웃하면 다시 동일하게 리다이렉트되는지 검증할 수
있습니다.

**Acceptance Scenarios**:

1. **Given** 로그인하지 않은 사용자가,
   **When** 메인 화면 URL로 직접 접근하면,
   **Then** 로그인 화면으로 자동 이동한다.
2. **Given** 로그인된 사용자가 메인 화면에서,
   **When** 로그아웃 버튼을 누르면,
   **Then** 로그인 화면으로 이동하고 인증 세션이 종료된다.
3. **Given** 로그아웃 직후,
   **When** 같은 브라우저에서 메인 화면 URL에 접근하면,
   **Then** 로그인 화면으로 이동한다(자동 로그인되지 않는다).
4. **Given** 로그인된 사용자가 메인 화면에서,
   **When** 페이지를 새로고침하면,
   **Then** 로그인이 유지되어 다시 인증할 필요가 없다.
5. **Given** 로그인된 사용자가,
   **When** 로그인 화면 URL에 접근하면,
   **Then** 메인 화면으로 자동 이동한다(이미 로그인된 사람이 로그인 폼을 다시 보지
   않도록).

**E2E Test Scenarios**:

- **접근 제어**: 비로그인 상태에서 메인, 캘린더, 통계 URL에 각각 직접 접근 시
  모두 로그인 화면으로 리다이렉트됨을 확인.
- **로그아웃 흐름**: 로그인 → 메인 → 로그아웃 클릭 → 로그인 화면 도착 → 메인 URL
  재접근 → 다시 로그인 화면으로 리다이렉트.
- **세션 지속성**: 로그인 → 새로고침 → 메인 유지 → 브라우저 탭 닫고 다시 열기 →
  로그인 유지(쿠키 만료 전까지).

### User Story 3 - Google 소셜 로그인 (Priority: P3)

사용자가 별도 이메일/비밀번호를 만들지 않고 본인의 Google 계정으로 한 번에 가입 및
로그인할 수 있습니다.

**Why this priority**: 접근성과 편의를 높이는 부가 기능입니다. US1, US2가 완비된
상태에서 추가됩니다. 없어도 시스템은 완전히 동작합니다.

**Independent Test**: 로그인 화면의 "Google로 계속하기" 버튼을 눌러 Google 인증 후
메인 화면 진입과 본인 할 일 목록 표시를 검증.

**Acceptance Scenarios**:

1. **Given** 로그인 화면에서,
   **When** "Google로 계속하기" 버튼을 누르면,
   **Then** Google 인증 페이지로 이동한다.
2. **Given** Google 인증이 성공하면,
   **When** 인증 콜백이 처리되면,
   **Then** 메인 화면으로 이동한다.
3. **Given** Google 계정으로 처음 로그인하는 사용자는,
   **When** 인증이 완료되면,
   **Then** 자동으로 계정이 생성되고 빈 할 일 목록의 메인 화면을 본다.
4. **Given** 이전에 Google 계정으로 가입한 사용자는,
   **When** 다시 Google로 로그인하면,
   **Then** 이전에 저장한 본인의 할 일 목록이 그대로 표시된다.

**E2E Test Scenarios**:

- "Google로 계속하기" 클릭 → Google 인증 동의 → 콜백 처리 → 메인 진입 확인.
- 동일 Google 계정으로 재로그인 → 이전 할 일 목록이 그대로 보임을 확인.

---

### Edge Cases

- 이미 가입된 이메일로 회원가입을 시도하면? → 명확한 한국어 오류 메시지("이미 가입된
  이메일입니다")와 로그인 안내가 표시된다.
- 비밀번호가 정책(최소 6자)을 만족하지 못하면? → 가입이 차단되고 정책 위반 사유가
  안내된다.
- 로그인 중 네트워크가 끊기면? → 작업이 일시적으로 차단되고 명확한 오류가 표시되며,
  재연결 시 자동 동기화된다.
- Google 인증을 사용자가 중간에 취소하면? → 로그인 화면으로 돌아오고 "로그인이
  취소되었습니다" 형태의 안내가 표시된다.
- 동일 사용자가 두 개의 브라우저(또는 기기)에서 동시에 로그인 후 한쪽에서 할 일을
  추가하면? → 페이지 새로고침 또는 재방문 시 다른 브라우저에도 반영된다.
- 세션 토큰이 만료되면? → 자동으로 로그인 화면으로 이동하고 재로그인 안내가 표시된다.
- 비로그인 상태에서 인증이 필요한 화면(캘린더, 통계 등)에 직접 접근하면? → 로그인
  화면으로 리다이렉트된다.

## Requirements *(mandatory)*

### Functional Requirements

**인증(Authentication):**

- **FR-001**: 시스템은 이메일과 비밀번호로 회원가입을 받아들여야 한다.
- **FR-002**: 시스템은 등록된 이메일/비밀번호로 로그인 인증을 처리해야 한다.
- **FR-003**: 시스템은 Google 계정을 통한 소셜 로그인 및 자동 회원가입을 지원해야
  한다.
- **FR-004**: 시스템은 로그인 세션이 페이지 새로고침과 브라우저 재시작(쿠키 만료 전까지)
  후에도 유지되도록 해야 한다.
- **FR-005**: 사용자는 어느 인증된 화면에서든 로그아웃할 수 있어야 한다. 로그아웃은
  인증 세션을 즉시 무효화해야 한다.

**라우팅/접근 제어:**

- **FR-006**: 시스템은 비로그인 상태에서 메인 화면 및 인증이 필요한 모든 하위 화면
  (캘린더, 통계 등) 접근 시 로그인 화면으로 자동 리다이렉트해야 한다.
- **FR-007**: 시스템은 로그인 상태에서 로그인 화면 URL 접근 시 메인 화면으로 자동
  이동시켜야 한다.

**데이터 격리:**

- **FR-008**: 사용자는 본인이 만든 할 일만 조회/수정/삭제할 수 있어야 한다. 다른
  사용자의 데이터에 대한 접근은 클라이언트와 서버 양쪽에서 모두 차단되어야 한다.
- **FR-009**: 시스템은 사용자별로 할 일을 클라우드 저장소에 영구 저장해야 하며,
  같은 계정으로 다른 기기에서 로그인하면 동일한 할 일 목록이 보여야 한다.

**오류/안내:**

- **FR-010**: 인증 실패 시(잘못된 비밀번호, 미가입 이메일, 중복 이메일 가입 시도 등)
  시스템은 사용자에게 명확한 한국어 오류 메시지를 표시해야 한다.
- **FR-011**: 시스템은 비밀번호 정책(최소 길이 등)을 사용자에게 명시하고, 정책 위반
  시 가입을 차단해야 한다.

**기존 기능 회귀:**

- **FR-012**: 인증 도입 후에도 기존 할 일 CRUD, 하위 작업 CRUD, 별표/완료 토글, 뷰
  필터링(today/upcoming/starred), 캘린더, 통계, 테마 토글이 변경 없이 정상 동작해야
  한다.

### Key Entities *(include if feature involves data)*

- **User**: 가입한 사용자의 신원. 이메일(이메일/비밀번호 가입 시) 또는 외부 식별자
  (Google 가입 시)를 포함하며, 다른 사용자와 구분되는 고유 ID를 가진다.
- **Task** (확장): 기존 할 일 엔티티는 이제 **소유자(User)** 와 연결된다. 소유자만 그
  할 일을 조회/수정/삭제할 수 있다. 기존 속성(제목, 메모, 완료 여부, 별표, 카테고리,
  기한, 하위 작업)은 유지된다.
- **Session**: 로그인한 사용자의 활성 세션. 페이지 새로고침이나 단기 브라우저 종료
  후에도 유지되며, 로그아웃 시 종료된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 새 사용자가 회원가입을 시작하여 첫 할 일을 추가 완료하기까지 **90초
  이내** 가능하다.
- **SC-002**: 서로 다른 두 사용자의 데이터가 **100% 격리**된다. 어떤 방법으로도 한
  사용자가 다른 사용자의 할 일을 조회/수정/삭제할 수 없다.
- **SC-003**: 비로그인 상태에서 인증이 필요한 화면에 접근 시 **100% 로그인 화면으로
  리다이렉트**된다(누락 0건).
- **SC-004**: 같은 계정으로 다른 기기(또는 시크릿 창)에서 로그인하면, 마지막으로
  저장된 할 일 목록이 **5초 이내**에 모두 표시된다.
- **SC-005**: 로그아웃 후 같은 브라우저에서 인증이 필요한 화면에 접근 시 **자동
  로그인되지 않으며 100% 로그인 화면이 표시**된다.
- **SC-006**: 인증 도입 전에 동작하던 모든 할 일 관련 기능이 인증 도입 후에도 **100%
  동작**한다(기존 기능 회귀 0건).
- **SC-007**: 회원가입/로그인/로그아웃 흐름에서 막힘이 발생한 경우 **5초 이내**에
  다음에 무엇을 해야 할지 명확한 안내가 표시된다.

## Assumptions

- 사용자는 수업 데모 및 친구 공유 시나리오이므로, **이메일 확인(verification email)은
  비활성화**한다. 회원가입 직후 즉시 로그인 상태가 된다.
- **비밀번호 정책은 표준 기본값**(최소 6자)을 사용한다.
- 기존 localStorage에 저장된 데모 할 일은 **새 사용자 데이터로 마이그레이션하지
  않는다**. 새 사용자는 빈 할 일 목록(또는 동일한 샘플 시드)으로 시작한다.
- **계정 삭제/탈퇴 기능은 v1 스코프에 포함하지 않는다**. 필요 시 별도 기능으로
  추가한다.
- **비밀번호 재설정(이메일 발송) 기능은 v1 스코프에 포함하지 않는다**. 비밀번호를
  잊으면 새 계정을 만들거나, 향후 별도 기능으로 추가한다.
- **모바일 전용 UI 최적화는 v1 스코프에 포함하지 않는다**(기존 디자인 그대로 유지).
- 친구에게 공유할 "링크"는 **각자 자신의 계정으로 가입/로그인하는 흐름**이며, 같은
  계정을 공유하거나 할 일을 협업 편집하는 시나리오가 아니다.
- Google OAuth용 클라이언트 ID와 시크릿은 외부 인증 공급자 콘솔에서 사전 설정되어
  있다고 가정한다(이는 운영 환경 구성의 문제이며 기능 명세의 범위가 아니다).
